# Monitoring

The lab uses the VictoriaMetrics Kubernetes stack for metrics, dashboards, and
alert evaluation.

## Components

Flux installs `victoria-metrics-k8s-stack` from the official VictoriaMetrics OCI
Helm repository:

```text
oci://ghcr.io/victoriametrics/helm-charts
chart: victoria-metrics-k8s-stack
version: 0.81.0
```

The stack currently includes:

- VictoriaMetrics Operator
- VMSingle
- VMAgent
- VMAlert
- VMAlertmanager
- Grafana
- kube-state-metrics
- prometheus-node-exporter

Grafana is exposed only through Tailscale:

```text
https://grafana-lab.bleak-banana.ts.net
```

Grafana requires login. The local generated admin material lives in:

```text
.secrets/grafana-admin.env
```

The cluster receives the credential through SOPS-encrypted manifests:

```text
clusters/lab/infrastructure/monitoring/grafana-admin.sops.yaml
clusters/lab/infrastructure/monitoring/grafana-helm-values.sops.yaml
```

`grafana-helm-values.sops.yaml` feeds `HelmRelease.spec.valuesFrom`, which lets
Helm set Grafana's generated admin Secret without committing the password in
plaintext.

## Host systemd Metrics

The Borg backup timer is monitored through Ubuntu's
`prometheus-systemd-exporter` package on each K3s host. Ansible manages the
service with:

```bash
make systemd-exporter
```

The exporter listens on port `9558` and is scoped to:

```text
borg-k3s-etcd-backup.service
borg-k3s-etcd-backup.timer
```

VictoriaMetrics scrapes the host exporters with a `VMStaticScrape` generated by
the Helm release.

## Initial Alerts

The lab-specific `VMRule` currently covers:

- node exporter down
- Kubernetes node disk pressure
- node filesystem under 10% free
- pod CrashLoopBackOff
- Flux GitOps Toolkit resource not ready
- Borg backup service failed
- Borg backup timer inactive
- Borg backup timer stale after a real timer trigger

The default Victoria/kube-prometheus-derived rules are also enabled by the
chart.

## Storage

`VMSingle` uses a `20Gi` `local-path` PVC. This is enough for the first platform
layer, but it is node-local storage. Treat it as operational telemetry, not
source-of-truth data.

## Verification

Check the stack:

```bash
kubectl -n monitoring get pods,svc,vmstaticscrape,vmrule,vmsingle,vmagent,vmalert,vmalertmanager
```

Check VMAgent targets:

```bash
kubectl -n monitoring port-forward svc/vmagent-vm 8429:8429
curl -fsS http://127.0.0.1:8429/targets
```

The `systemd-exporter` job should show `3/3 up`.
